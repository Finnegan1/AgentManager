The **MCP gateway** is the heart of Agent Manager. It's a centralized HTTP daemon that aggregates multiple downstream MCP servers and your local skill library into a single connection point for your AI clients.

## The Problem It Solves

Without a gateway, your AI client needs a separate connection for each MCP server you want to use. Managing 5, 10, or more individual server connections becomes unwieldy — each with its own configuration, lifecycle, and potential for naming conflicts.

The gateway solves this by acting as a **single HTTP endpoint** that your clients connect to. Behind the scenes, it connects to all your downstream servers and proxies their capabilities transparently.

## How It Works

## Multi-Session Architecture

The gateway runs as a single process, but each connected AI client gets its own isolated **session**. This is because the MCP SDK's `Server` class supports exactly one transport — so each client connection creates a new `Server` + `StreamableHTTPServerTransport` pair.

Sessions are identified by the `mcp-session-id` HTTP header, which is assigned during the MCP `initialize` handshake.

### Session Lifecycle

1. **Client sends POST /mcp** — The gateway creates a new `Server` + `Transport` pair
2. **Initialize handshake** — The MCP protocol's `initialize` request completes, and a session ID is assigned
3. **Project root detection** — The gateway queries the client's project root (via `roots/list` or `X-Project-Root` header)
4. **Project attachment** — If a project root is found, project-scoped downstream connections are attached to the session
5. **Normal operation** — The client uses tools, resources, and prompts via the session
6. **Cleanup** — Sessions are removed on disconnect or after 30 minutes of inactivity

## Server Scoping

Downstream servers have a **scope** that determines how they are managed:

- **`"global"`** (default) — Started once and shared across all sessions. Use for APIs, databases, web services, and other context-independent servers.
- **`"project"`** — Started per project root with the project's working directory as `cwd`. Sessions sharing the same project root share these connections. Use for filesystem tools, linters, Git servers, and other context-dependent tools.

See [Downstream Servers — Scoping](/docs/concepts/downstream-servers#scoping) for more details.

## Connecting Your AI Client

Configure your AI client to connect to the gateway's HTTP endpoint. For Claude Code, add to `.mcp.json`:

```json
{
  "mcpServers": {
    "skill-gateway": {
      "type": "http",
      "url": "http://127.0.0.1:24842/mcp"
    }
  }
}
```

<Callout>
  The gateway must be running before your AI client connects. Start it with `bun run dev:server` in the `apps/mcp-gateway/` directory.
</Callout>

## Namespacing

To prevent name collisions between downstream servers, the gateway prefixes all proxied items with the **server key**:

| Type | Original Name | Gateway Name |
|---|---|---|
| Tool | `run_query` on server `db` | `db__run_query` |
| Resource | `schema://tables` on server `db` | `db://schema://tables` |
| Prompt | `analyze` on server `ai` | `ai__analyze` |

The double-underscore `__` separator for tools and prompts, and the `serverKey://` prefix for resources, ensure that items from different servers never collide.

## Native Tools

The gateway exposes five built-in tools that are always available, regardless of downstream server connections:

| Tool | Description |
|---|---|
| `skills__list` | List all skills with metadata |
| `skills__search` | Search skills by query |
| `skills__get` | Get full skill content by ID |
| `gateway__list_servers` | List all downstream servers and their connection status |
| `gateway__server_status` | Get detailed status of a specific downstream server |

## Hot-Reload

The gateway watches the configuration file (`~/.agent-manager/config.json`) for changes. When you add, remove, enable, disable, or modify a downstream server in the desktop app:

1. The config file is updated on disk
2. The gateway's `ConfigStore` detects the change (debounced at 200ms)
3. The `ProxyManager` diffs the old and new configurations
4. New servers are connected, removed servers are disconnected, changed servers are reconnected
5. The tool/resource/prompt registries are updated
6. A `listChanged` notification is broadcast to **all connected sessions**

This means you never need to restart the gateway to apply configuration changes.

## Auto-Reconnect

If a downstream server connection drops, the gateway automatically attempts to reconnect with exponential backoff — up to 10 retries with a maximum delay of 30 seconds between attempts.

## HTTP Endpoints

| Endpoint | Method | Description |
|---|---|---|
| `/mcp` | POST | MCP requests (initialize, tool calls, etc.) |
| `/mcp` | GET | SSE stream for server-to-client notifications |
| `/mcp` | DELETE | Close a session |
| `/health` | GET | Health check with PID, uptime, and session count |
