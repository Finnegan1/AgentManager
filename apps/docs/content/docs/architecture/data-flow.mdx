Understanding the data flows in Agent Manager helps you reason about how changes propagate through the system.

## HTTP Request Flow

When an AI client connects to the gateway:

```
AI Client sends POST http://127.0.0.1:24842/mcp
    │
    ▼
HTTP Server receives request
    │
    ├── Has mcp-session-id header with known session?
    │   YES → Route to existing session's transport
    │   NO  → Create new session (see Session Lifecycle below)
    │
    ▼
StreamableHTTPServerTransport.handleRequest(req, res)
    │
    ▼
Session's Server processes the MCP request
    │
    ▼
Response sent back to AI client
```

## Session Lifecycle

When a new AI client connects for the first time:

```
POST /mcp (no session ID, contains "initialize" request)
    │
    ▼
SessionManager.createSession()
    │  Creates new Server + StreamableHTTPServerTransport pair
    │  Registers request handlers (tools, resources, prompts)
    │
    ▼
Transport handles the initialize handshake
    │  Session ID assigned (UUID)
    │  Client receives mcp-session-id header
    │
    ▼
onSessionInitialized callback fires
    │
    ▼
Gateway queries client for project root
    │  (via roots/list or X-Project-Root header)
    │
    ▼
SessionManager.attachProjectRoot(sessionId, root)
    │  Acquires project ProxyManager from ProjectConnectionPool
    │  Re-registers handlers to include project-scoped tools
    │  Sends listChanged notification to client
    │
    ▼
Client now has access to:
    ├── Native tools (skills, gateway)
    ├── Global-scope downstream tools
    └── Project-scope downstream tools (for their project root)
```

Sessions are cleaned up:
- When the client disconnects (transport `onclose`)
- After 30 minutes of inactivity (periodic cleanup every 60 seconds)

## Skill Write Flow

When a user creates or edits a skill in the desktop app:

```
User edits in CodeMirror editor
    │
    ▼
React frontend calls saveSkill(id, content)
    │
    ▼
Tauri IPC → Rust backend
    │
    ▼
Rust writes ~/.agent-manager/skills/{id}.md
    │
    ▼
Frontend reloads skill list
```

The MCP gateway reads skills on demand (when `skills__list`, `skills__search`, or `skills__get` are called), so new skills are available immediately without any restart or notification.

## Configuration Change Flow

When a user modifies server configuration in the desktop app:

```
User adds/edits/toggles a server
    │
    ▼
React frontend calls saveConfig(config)
    │
    ▼
Tauri IPC → Rust backend
    │
    ▼
Rust atomically writes ~/.agent-manager/config.json
    │                    (write to temp file, then rename)
    ▼
Gateway's ConfigStore detects file change (fs.watch)
    │                    (debounced 200ms)
    ▼
ConfigStore validates new config (Zod schema)
    │
    ▼
ConfigStore emits "changed" event
    │
    ▼
Global ProxyManager diffs old ↔ new config (global-scope servers)
    │
    ├── New servers → create DownstreamConnection, connect
    ├── Removed servers → disconnect, clean up
    ├── Changed servers → disconnect old, reconnect new
    └── Disabled servers → disconnect, remove from registries
    │
    ▼
Each project ProxyManager also receives the change event
    │  (handles project-scope server changes independently)
    │
    ▼
Registries updated (tools, resources, prompts)
    │
    ▼
SessionManager broadcasts "listChanged" to ALL connected sessions
```

<Callout>
  Config writes use an atomic pattern (write to temp file, then rename) to prevent partial writes from crashing the gateway.
</Callout>

## Tool Proxying Flow

When an AI client calls a proxied tool through the gateway:

```
AI Client calls tool "db__run_query" with args
    │
    ▼
Session's Server receives CallToolRequest
    │
    ▼
Check project proxy first (project-scoped tools take priority)
    │
    ├── Found in project proxy? → forward to project downstream
    └── Not found? → check global proxy
    │
    ▼
ToolRegistry.resolveToolCall("db__run_query")
    │
    ├── Parses namespace: server="db", tool="run_query"
    └── Looks up DownstreamConnection for "db"
    │
    ▼
DownstreamConnection.callTool("run_query", args)
    │
    ▼
Downstream MCP server "db" executes the tool
    │
    ▼
Result flows back through the gateway to the AI client
```

The same pattern applies to resource reads and prompt requests — the gateway resolves the namespace, finds the correct downstream connection, and forwards the request. Project-scoped servers take priority over global servers when resolving names.

## Gateway Status Polling

The desktop app monitors the gateway's status through a shared file:

```
MCP Gateway HTTP Daemon (running on port 24842)
    │
    │  Every 5 seconds:
    ▼
Writes ~/.agent-manager/status.json
    │  {
    │    "pid": 12345,
    │    "startedAt": "2026-03-22T10:00:00Z",
    │    "port": 24842,
    │    "url": "http://127.0.0.1:24842/mcp",
    │    "activeSessions": 3,
    │    "servers": {
    │      "db": { "connected": true, "toolCount": 5, ... },
    │      "ai": { "connected": false, "error": "timeout", ... }
    │    }
    │  }
    │
    ▼
Desktop App polls every 5 seconds:
    │
    ▼
Tauri IPC → Rust reads status.json
    │
    ▼
Rust verifies PID is still running
    │  (Unix: kill -0; Windows: tasklist)
    │
    ├── PID alive → return status as-is
    └── PID dead → clean up stale file, return "not running"
    │
    ▼
React frontend applies disconnect stabilization
    │  (requires 2 consecutive "disconnected" polls
    │   before marking a server as disconnected)
    │
    ▼
Dashboard UI updates
```

<Callout type="warn">
  The disconnect stabilization logic prevents the UI from flickering when a downstream server briefly disconnects and reconnects. A server is only shown as disconnected after two consecutive polls report it as down.
</Callout>

## Downstream Server Connection Flow

When a new downstream server is added and enabled:

```
ProxyManager creates DownstreamConnection
    │
    ▼
Connection creates appropriate transport:
    ├── stdio: spawn child process (cwd overridden for project-scope)
    ├── SSE: HTTP connection with EventSource
    └── streamable-http: HTTP connection
    │
    ▼
MCP Client connects to downstream server
    │
    ▼
Capability discovery:
    ├── listTools() → tools registered in ToolRegistry
    ├── listResources() → resources registered in ResourceRegistry
    └── listPrompts() → prompts registered in PromptRegistry
    │
    ▼
Connection marked as "connected"
    │
    ▼
SessionManager broadcasts "listChanged" to all connected sessions
```

If the connection fails, exponential backoff retry kicks in — up to 10 attempts with a maximum 30-second delay between retries.
